A modern industrial servo drive — for a robot joint, a CNC spindle, a packaging-machine axis — runs **field-oriented control** (FOC) on a Cortex-M, sampling motor currents at the precise instant the PWM is at midpoint, mixing the αβ-to-dq transforms, running two cascaded PI loops, and updating PWM duty cycles **all within 50 µs every iteration**. The MCU has been built specifically for this since the STM32F3 generation; it's where ARM Cortex-M flexes.

## Field-Oriented Control in 90 seconds

A PMSM (Permanent Magnet Synchronous Motor) is most efficient when you apply current **90° ahead of the rotor's magnetic axis**. Doing that requires knowing where the rotor *is* and applying a precise current at the right phase.

```
         3-phase currents iA, iB, iC (measured)
                       │
                       ▼
         ┌─────────────────────────┐
         │ Clarke transform        │   iA, iB, iC → iα, iβ   (3→2)
         └─────────────┬───────────┘
                       ▼
         ┌─────────────────────────┐
         │ Park transform          │   iα, iβ, θ → id, iq   (rotating frame)
         └─────────────┬───────────┘
                       ▼
         ┌─────────────────────────┐
         │ PI controller d-axis    │   id_ref - id  → ud
         │ PI controller q-axis    │   iq_ref - iq  → uq
         └─────────────┬───────────┘
                       ▼
         ┌─────────────────────────┐
         │ Inverse Park            │   ud, uq, θ → uα, uβ
         └─────────────┬───────────┘
                       ▼
         ┌─────────────────────────┐
         │ Space-vector PWM        │   uα, uβ → 3× PWM duties → IGBT/MOSFETs
         └─────────────────────────┘
```

Every one of these blocks is ~10–30 multiplies. Total per iteration: ~150 floating-point ops. At 16 kHz iteration rate, that's 2.4 Mflops of motor control — comfortable on a Cortex-M4F at 170 MHz, easy on a Cortex-M7 at 480 MHz.

The cascaded position-velocity-current loop structure:

```
position_setpoint → [P controller] → velocity_setpoint
                                        │
                          velocity ←────┤
                                        ▼
                                    [PI controller] → iq_setpoint
                                                          │
                                            iq, id ←──────┤
                                                          ▼
                                                      FOC inner loop
                                                          │
                                                          ▼
                                                       3× PWM
```

Outer loops run slower (4 kHz velocity, 1 kHz position) because mechanical dynamics are slower than electrical.

## Current sensing — the timing-critical part

The 3-phase currents are measured with shunt resistors in the inverter's low-side legs. To get accurate readings you must **sample at PWM midpoint** — the only moment when all three low-side MOSFETs are conducting and the shunt currents represent phase current cleanly.

```
   PWM (centre-aligned):
       ▲▲▲                  ▲▲▲                  ▲▲▲
   ___╱   ╲_______________╱   ╲_______________╱   ╲___
      
   ADC trigger ↑ here       ↑ here              ↑ here
   (PWM midpoint = ARR/2 of the timer counter)
```

The peripheral arrangement: **TIM1 / HRTIM generates the centre-aligned PWM**, **TRGO at compare match** triggers the **ADC**, **ADC sequence** samples 3 shunts (plus DC link voltage and temperature), **EOC interrupt** fires the FOC ISR which reads the 5 values out of the ADC regular sequence.

```c
// FOC ISR — fires at every PWM peak, ~62 µs at 16 kHz
__attribute__((section(".ccmram.text")))  // place in CCM RAM for determinism
void TIM1_UP_IRQHandler(void) {
    // ~ 2 cycles to enter, registers stacked
    TIM1->SR = 0;  // clear

    // Read 3 ADC channels (already converted by hardware)
    int16_t ia = ADC1->DR;   // phase A current
    int16_t ib = ADC2->DR;   // phase B current
    int16_t ic = ADC3->DR;   // phase C current (or compute: -iA-iB)

    // Read encoder for theta
    uint16_t theta_raw = TIM2->CNT;
    float theta = theta_raw * THETA_SCALE;

    // Clarke + Park
    float ialpha = (2*ia - ib - ic) * (1.0f/3.0f);
    float ibeta  = (ib - ic) * INV_SQRT3;
    float c = __builtin_arm_cordic_cos(theta);
    float s = __builtin_arm_cordic_sin(theta);
    float id =  c*ialpha + s*ibeta;
    float iq = -s*ialpha + c*ibeta;

    // PI controllers
    float ud = pi_d_update(id_ref - id);
    float uq = pi_q_update(iq_ref - iq);

    // Inverse Park
    float ualpha = c*ud - s*uq;
    float ubeta  = s*ud + c*uq;

    // Space-vector modulation → 3 PWM duties
    svpwm(ualpha, ubeta, &TIM1->CCR1, &TIM1->CCR2, &TIM1->CCR3);

    // Done — total ≈ 35 µs out of 62 µs budget
}
```

On the G4 / G474 the **CORDIC** is a hardware coprocessor that does sin/cos in ~20 cycles instead of 100+ via a Taylor expansion. The **FMAC** does block filter operations. Both fed via DMA-coupled mailboxes for absolute-best timing.

## Safe Torque Off (STO)

In any drive sold for industrial use, **STO** (Safe Torque Off) is mandatory. It's a hardware input — when asserted, **the gate drivers are unconditionally disabled in hardware**, regardless of firmware state. The motor coasts to a stop.

```
   ┌─────────┐         ┌──────────────┐
   │ Safety  │── STO+ ─▶│ Gate driver │── PWM_A_HI ─▶ MOSFET
   │ relay   │── STO- ─▶│ enable_pin   │
   └─────────┘         └──────────────┘

   When STO is open (de-energised):
     - Gate driver outputs are forced LOW (off)
     - Firmware cannot turn the MOSFETs on no matter what it tries
```

This is **separate from any software safety**. It's two redundant hardware inputs that physically gate the gate driver supplies. SIL-3 compliance requires this.

## Communication: CANopen and EtherCAT

The drive is one node on a fieldbus. The two most common protocols:

**CANopen** (over CAN-FD): older, widely supported, ~1 ms cycle time.
- PDOs (Process Data Objects) carry the real-time data: position commands, status.
- SDOs (Service Data Objects) carry configuration: PI gains, motor parameters.
- Synchronization via SYNC message.

**EtherCAT**: modern, 100 Mbps Ethernet variant with hard 32 µs cycle times across hundreds of axes.
- Distributed Clocks for synchronized motion across axes.
- Each axis is a "slave" with an ESC (EtherCAT Slave Controller) ASIC alongside the MCU.
- Drives sit on a single-cable daisy-chain.

For modern multi-axis machines (robotics, CNC), EtherCAT has largely won. Each drive has an additional ESC chip (Beckhoff ET1100, Microchip LAN9252) that handles the Ethernet/EtherCAT side and presents a memory-mapped DPRAM to the Cortex-M for shared data.

## What's hard about this

- **Tuning** — the PI gains for current and velocity loops depend on motor inductance, back-EMF constant, inertia, and load. Auto-tuning algorithms exist but production-grade tuning is still half-art.
- **EMC** — switching 50 A at 20 kHz radiates *a lot*. Layout, snubbers, gate driver slew rate all matter. Many drive designs spend more time in EMC chambers than in writing firmware.
- **Sensor faults** — encoder failures, current-sensor offset drift, hall-sensor glitches must all be detected and handled without commanding torque excursions.
- **Cogging compensation** — to get smooth motion at low speed, firmware compensates for motor cogging torque via a feed-forward table.

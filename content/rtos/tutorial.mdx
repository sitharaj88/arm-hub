Most engineers learn an RTOS the wrong way. They read three blog posts, port the demo project, get a blinking LED, then graft a queue and a semaphore onto whatever code they happen to be writing. Three months later, they have a system that mostly works, occasionally locks up, and that nobody on the team really understands. The mental model never gelled because the tutorial skipped the parts that matter: why an RTOS at all, what `xTaskCreate` *actually* allocates, why the SysTick handler has to be the lowest-priority interrupt in the system, and what happens to a hardware ISR that calls `xQueueSend` instead of `xQueueSendFromISR`.

This tutorial is the version I wish I had been handed in 2012. Ten lessons, built on real silicon, in linear order. By the end you will have written, debugged, traced, and reasoned about a small but complete FreeRTOS application running on a Cortex-M4 board you can buy for fifteen dollars. The mental model will transfer to Zephyr, ThreadX, RT-Thread, and any future RTOS you encounter, because the underlying ideas — pre-emption, priority, blocking, IPC, the Cortex-M port internals — are universal. Only the API names change.

## Why this tutorial and not the FreeRTOS docs

The official FreeRTOS documentation is reference material. It tells you what `xQueueCreate` does, not when to reach for a queue versus a stream buffer versus a task notification. It tells you what `configMAX_SYSCALL_INTERRUPT_PRIORITY` is, not what catastrophe befalls you when you set it wrong. Reference docs are organised by API surface; tutorials are organised by the order in which questions arise in your head as you build something. This is a tutorial. We will keep returning to the same six or seven core ideas — priority, blocking, context switch, ISR safety, memory layout, predictability, observability — and look at them from progressively more interesting angles.

The other reason: the FreeRTOS demos compile, but they do not teach. Most of them assume you already know what a task is, what a tick is, and what an idle hook is for. We start much earlier.

## Philosophy — one RTOS deeply

You only need to learn one RTOS deeply. The vocabulary maps cleanly across the field:

- FreeRTOS `xTaskCreate` is Zephyr `K_THREAD_DEFINE`, ThreadX `tx_thread_create`, RT-Thread `rt_thread_create`.
- FreeRTOS `xQueueSend` is Zephyr `k_msgq_put`, ThreadX `tx_queue_send`, RT-Thread `rt_mq_send`.
- FreeRTOS `xSemaphoreTake` is Zephyr `k_sem_take`, ThreadX `tx_semaphore_get`, RT-Thread `rt_sem_take`.
- Priority inheritance, tickless idle, software timers, ISR-safe API variants — every modern RTOS has them. Only the spellings differ.

We picked FreeRTOS because:

- It is MIT-licensed, free to use commercially, and shipped on more Cortex-M designs than every other RTOS combined.
- Its kernel is tiny (about 9000 lines of C) — you can read every line.
- The port for ARM Cortex-M is a model of clarity. PendSV for context switch, SysTick for tick, SVC to bootstrap. We will dissect it in lesson 2.
- Amazon's stewardship since 2017 has stabilised the release process without changing the API. Code from 2012 still compiles against the current kernel.

If you finish this tutorial and then move to Zephyr because your employer requires it, you will spend a weekend learning the Kconfig + device-tree workflow and then feel right at home. The kernel concepts are the same.

## Prerequisites

You should know C — pointers, function pointers, structs, the difference between `static` and `extern`, basic preprocessor. If any of that feels shaky, work through the [C tutorial](/docs/programming/c-tutorial/) first; it is calibrated to set you up for embedded work.

You should also be comfortable bringing up a bare-metal project on Cortex-M — that is, you can write a startup file, configure a clock tree, get a blinking LED working without an RTOS. If you cannot, our [Cortex-M lineup](/docs/cortex-m/lineup/) page and the [exception model](/docs/architecture/exception-model/) page give the background; lesson 2 will not re-derive any of that.

Hardware: an [STM32F411 Nucleo-64](https://www.st.com/en/evaluation-tools/nucleo-f411re.html) is the reference. Any Cortex-M3, M4, M7, or M33 board works with minor adjustments. The Nucleo has its built-in ST-Link debugger, a UART-over-USB bridge mapped to USART2, an LED on PA5, and a button on PC13 — enough to do everything we need.

## How to use this tutorial

Read each lesson cover-to-cover before touching code. The lessons are written to be understood in one sitting; the code is at the end. Then build the code yourself — copy it, do not paste it. Typing the call sequences is part of how the API gets into your fingers.

Every lesson ends with two or three **Exercises**. They are not optional. The answers are in collapsed callouts at the end of each lesson, but resist looking until you have either solved the exercise on hardware or convinced yourself you know exactly why it works. Real embedded debugging is "stare at the symptom and form a hypothesis"; the exercises are calibrated to that.

If something behaves unexpectedly on your board, do not move on. RTOS bugs are vastly easier to diagnose in a five-task system than in a twenty-task system. Fix it now.

<Callout type="tip" title="Use a logic analyser if you can">
A cheap USB logic analyser (Saleae clone, $10) plugged into two GPIOs you toggle from your tasks is the single most valuable debug tool for this tutorial. You will see context switches happen, you will see ISR latency, you will see priority inversion. Print-based debugging hides all of that.
</Callout>

<Callout type="warn" title="Do not start with an RTOS in your day job">
If you are an embedded engineer reading this tutorial because your team is "considering an RTOS for the next project" — pause. Lesson 1 walks through the decision criteria honestly. RTOSes are a tool, not a virtue. Some of the best firmware ever shipped never had one.
</Callout>

## Conventions used throughout

Code is in C unless otherwise stated. Shell commands assume bash on Linux or macOS; Windows users can use WSL or the equivalent native tooling. Snippets that need vendor HAL calls (clock init, GPIO setup) abstract them as `SystemClock_Config()` and `gpio_init()` — the focus is the FreeRTOS API, not the vendor's clock-tree wizard.

Stack sizes are given in *words*, not bytes, because that is FreeRTOS's convention. `configMINIMAL_STACK_SIZE = 128` means 128 × 4 = 512 bytes on Cortex-M. We will repeat this every time it matters.

Priority values follow FreeRTOS's "higher number is higher priority" convention. This is the opposite of NVIC priorities, which are "lower number is higher urgency." Lesson 5 explains why both conventions exist and how to keep them straight.
